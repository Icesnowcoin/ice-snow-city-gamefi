## 1. 核心架构模型

### 1.1 两层交互模型

```
┌─────────────────────────────────────────────────────────────────┐
│                    Ice Snow City 交互架构                        │
└─────────────────────────────────────────────────────────────────┘

第 1 层: 游戏内部交互 (Game Internal)
├─ 发生位置: 游戏客户端 + 游戏服务器
├─ 参与者: 玩家、NPC、游戏系统
├─ 交互方式: HTTP/WebSocket (tRPC)
├─ 数据存储: MySQL 数据库
├─ 特点: 快速、实时、无延迟
├─ 示例:
│   ├─ 玩家完成任务 → 立即显示奖励
│   ├─ 玩家购买商品 → 立即添加到背包
│   ├─ 玩家与 NPC 交互 → 立即显示对话
│   └─ 玩家查看余额 → 立即显示最新余额
└─ 频率: 非常高频 (每秒数十次)

第 2 层: 区块链交互 (Blockchain)
├─ 发生位置: BSC 区块链
├─ 参与者: 游戏金库钱包、玩家钱包、NPC 钱包
├─ 交互方式: 智能合约调用 (Web3.js)
├─ 数据存储: 区块链 (不可篡改)
├─ 特点: 缓慢、异步、有延迟 (12-15 秒)
├─ 示例:
│   ├─ ISC 代币转账
│   ├─ 交易记录永久化
│   ├─ 资产所有权证明
│   └─ 跨平台转账
└─ 频率: 低频 (每分钟数次)

两层的关系:
游戏内部交互 ← 同步 → 区块链交互
  ↓
玩家立即看到结果 (游戏内部)
  ↓
结果最终被记录到区块链 (异步)
```

## 2. 详细的交互流程

### 2.1 玩家完成任务的完整流程

```
时间线:
┌─────────────────────────────────────────────────────────────┐

T0: 玩家点击"完成任务"按钮
  ↓
T0.1: 游戏客户端发送请求到游戏服务器
  └─ 请求: POST /trpc/playerISC.completeTask
  └─ 数据: { taskId: "task_001" }
  └─ 延迟: < 100ms

T0.2: 游戏服务器验证任务
  ├─ 检查任务是否存在
  ├─ 检查任务是否已完成
  ├─ 检查玩家是否满足条件
  └─ 延迟: < 50ms

T0.3: 游戏服务器计算奖励
  ├─ 基础奖励: 100 ISC
  ├─ 关系加成: 10% (如果有)
  ├─ 最终奖励: 110 ISC
  └─ 延迟: < 10ms

T0.4: 游戏服务器创建交易记录 (待处理)
  ├─ 在数据库中插入交易记录
  ├─ 状态: "pending"
  ├─ txId: "tx_1234567890"
  └─ 延迟: < 50ms

T0.5: 游戏服务器更新玩家余额 (数据库)
  ├─ 读取当前余额: 500 ISC
  ├─ 计算新余额: 500 + 110 = 610 ISC
  ├─ 更新数据库
  └─ 延迟: < 50ms

T0.6: 游戏服务器返回结果给客户端
  ├─ 返回新余额: 610 ISC
  ├─ 返回交易 ID: tx_1234567890
  ├─ 返回状态: "success"
  └─ 延迟: < 100ms

T1 (总耗时 < 500ms): 玩家看到结果
  ├─ 屏幕显示: "获得 110 ISC！"
  ├─ 余额更新: 500 ISC → 610 ISC
  ├─ 任务标记为已完成
  └─ 用户体验: 即时反馈 ✅

─────────────────────────────────────────────────────────────

T2 (后台异步处理): 游戏服务器执行区块链转账
  ├─ 调用智能合约 transfer() 函数
  ├─ 参数: 
  │   ├─ from: 游戏金库钱包 (0xGameTreasury...)
  │   ├─ to: 玩家钱包 (0xPlayer...)
  │   └─ amount: 110 ISC
  ├─ 交易发送到 BSC 区块链
  └─ 延迟: 立即发送

T3 (12-15 秒后): 区块链确认交易
  ├─ 交易被打包到区块
  ├─ 交易获得确认
  ├─ txHash: 0xabcdef1234567890...
  └─ 区块链状态: 不可篡改 ✅

T4 (后台处理): 游戏服务器更新交易记录
  ├─ 读取交易记录 (txId: tx_1234567890)
  ├─ 更新状态: "pending" → "success"
  ├─ 添加 txHash: 0xabcdef1234567890...
  ├─ 添加完成时间: T3
  └─ 延迟: < 50ms

T5 (最终状态): 系统完全同步
  ├─ 游戏数据库: 玩家余额 610 ISC ✅
  ├─ 区块链: ISC 转账已确认 ✅
  ├─ 审计日志: 交易已记录 ✅
  └─ 玩家钱包: 余额增加 110 ISC ✅

总时间:
├─ 玩家感知: < 500ms (即时)
├─ 区块链确认: 12-15 秒 (后台)
└─ 完全同步: 15-20 秒 (后台)
```

### 2.2 与玩家钱包的交互

```
玩家钱包交互只发生在两个场景:

场景 1: 玩家主动提现
  ├─ 玩家在游戏内点击"提现"
  ├─ 游戏验证余额
  ├─ 游戏执行区块链转账 (游戏金库 → 玩家钱包)
  ├─ 玩家钱包接收 ISC
  ├─ 玩家可以在 MetaMask/钱包中查看
  └─ 这是玩家主动的、可选的操作

场景 2: 玩家主动充值
  ├─ 玩家在游戏内点击"充值"
  ├─ 游戏生成充值地址
  ├─ 玩家从自己的钱包转账 ISC 到游戏金库
  ├─ 游戏监听区块链事件
  ├─ 游戏确认充值交易
  ├─ 游戏更新玩家余额
  └─ 这是玩家主动的、可选的操作

日常游戏活动 (完成任务、购买商品等):
  ├─ 不需要玩家钱包参与
  ├─ 完全在游戏内部进行
  ├─ 玩家钱包无需签名或授权
  ├─ 玩家无需打开 MetaMask
  ├─ 玩家无需支付 Gas 费
  └─ 用户体验: 完全无感知 ✅
```

## 3. 架构对比

### 3.1 游戏内部交互 vs 区块链交互

| 特性 | 游戏内部交互 | 区块链交互 |
|------|-----------|---------|
| **发生位置** | 游戏服务器 + 数据库 | BSC 区块链 |
| **参与者** | 玩家、NPC、游戏系统 | 钱包地址、智能合约 |
| **速度** | 极快 (< 500ms) | 较慢 (12-15 秒) |
| **频率** | 非常高频 (每秒数十次) | 低频 (每分钟数次) |
| **延迟** | 无延迟 | 有延迟 |
| **成本** | 无成本 | 需要 Gas 费 |
| **用户感知** | 即时反馈 | 后台异步 |
| **数据存储** | MySQL 数据库 | 区块链 (不可篡改) |
| **用途** | 日常游戏活动 | 资产证明、审计、跨平台 |
| **示例** | 完成任务、购买商品 | 提现、充值、转账 |

### 3.2 交互流程对比

```
游戏内部交互流程:
玩家操作 → 游戏验证 → 计算结果 → 更新数据库 → 返回结果 → 玩家看到结果
  ↓
  └─ 耗时: < 500ms
  └─ 用户体验: 即时反馈 ✅

区块链交互流程:
玩家操作 → 游戏验证 → 构建交易 → 发送到区块链 → 等待确认 → 更新数据库 → 返回结果
  ↓
  └─ 耗时: 12-15 秒
  └─ 用户体验: 后台异步 ✅
```

## 5. 数据流向

### 5.1 完整的数据流向图

```
┌─────────────────────────────────────────────────────────────┐
│                    数据流向图                                │
└─────────────────────────────────────────────────────────────┘

玩家操作
  ↓
游戏客户端 (React)
  ├─ 发送 tRPC 请求
  └─ 显示结果

  ↓
游戏服务器 (Express + tRPC)
  ├─ 验证请求
  ├─ 执行业务逻辑
  ├─ 更新数据库
  └─ 返回结果

  ↓
MySQL 数据库
  ├─ 存储玩家账户
  ├─ 存储交易记录
  ├─ 存储游戏数据
  └─ 提供快速查询

  ↓
区块链 (后台异步)
  ├─ 执行 ISC 转账
  ├─ 记录交易
  └─ 提供不可篡改的证明

  ↓
玩家钱包 (可选)
  ├─ 接收 ISC (提现时)
  ├─ 发送 ISC (充值时)
  └─ 查看余额

  ↓
审计日志
  ├─ 记录所有操作
  ├─ 加密存储
  └─ 用于审计和追踪
```

### 5.2 交易的三个状态

```
交易状态机:

pending (待处理)
  ├─ 交易已在数据库中创建
  ├─ 玩家已看到结果
  ├─ 区块链交易正在发送
  └─ 持续时间: 0-15 秒

success (成功)
  ├─ 区块链交易已确认
  ├─ txHash 已记录
  ├─ 数据库已更新
  └─ 持续时间: 15+ 秒

failed (失败)
  ├─ 区块链交易失败
  ├─ 玩家余额已回滚
  ├─ 玩家收到失败通知
  └─ 持续时间: 15+ 秒

转换流程:
pending → (等待 12-15 秒) → success
pending → (交易失败) → failed
```

## 6. 关键问题解答

### Q1: 玩家完成任务后，ISC 立即到账吗？

**A**: 是的，立即到账。
- 游戏内部: 立即更新玩家余额 (< 500ms)
- 区块链: 后台异步处理 (12-15 秒后确认)
- 用户体验: 完全无感知，看不到延迟

### Q2: 玩家需要签名或授权吗？

**A**: 不需要。
- 日常游戏活动: 无需签名，无需授权
- 只有提现和充值时: 需要区块链交易 (但由游戏服务器执行)
- 玩家无需打开 MetaMask

### Q3: 玩家需要支付 Gas 费吗？

**A**: 不需要。
- 日常游戏活动: 无 Gas 费
- 提现和充值: 由游戏金库支付 Gas 费
- 玩家完全无需支付

### Q4: 如果区块链交易失败怎么办？

**A**: 系统会自动处理。
- 玩家余额立即回滚
- 交易状态标记为 "failed"
- 玩家收到失败通知
- 可以重试

### Q5: 玩家的 ISC 存储在哪里？

**A**: 三个地方同时存储。
- 区块链: 真实存储 (不可篡改)
- 数据库: 缓存存储 (快速查询)
- 玩家钱包: 可选存储 (提现后)
